# Dotfiles Design: Themes, Claude Config, and Work Machine Sync

Date: 2026-03-07

## Goals

1. Solarized Light/Dark everywhere, switching by **time of day** (not system preference)
2. Claude config files (`CLAUDE.md`, `settings.json`, `mcp.json`) managed in dotfiles
3. Clear process for syncing work machine — capturing drift and reconciling custom values
4. Remote server treated as a first-class AI coding environment (not an afterthought)

---

## 1. Time-of-Day Theming

### Approach

A central script `~/.local/bin/theme-current` outputs `light` or `dark` based on time. All tools source or call this at startup. A cron job (or launchd on macOS) runs the switch script at transition times to update running tools.

**Default schedule:** light 07:00–18:00, dark 18:00–07:00.

### Per-tool implementation

| Tool | Mechanism |
|------|-----------|
| **Ghostty** | Already supports `light:Builtin Solarized Light,dark:Builtin Solarized Dark`. Change to system-preference _or_ call `ghostty +set-config theme=...` via script |
| **Tmux** | Source a theme file chosen by the script. `tmux source-file` reloads at switch time |
| **Neovim** | `auto-dark-mode.nvim` plugin with custom `get_mode` that calls `theme-current`. Applies Solarized colorscheme via `NeoSolarized` or `solarized.nvim` |
| **VS Code** | `vscode-auto-dark-mode` extension _or_ a settings template that sets `workbench.colorTheme` based on role; can't easily auto-switch without an extension |
| **Starship** | Colors are fixed — Solarized-compatible palette works in both modes |
| **Zsh prompt** | Same — pick colors that read well on both Solarized Light and Dark |

### Switch script skeleton

```sh
#!/bin/sh
# ~/.local/bin/theme-switch  (called by cron at 07:00 and 18:00)
THEME=$(~/.local/bin/theme-current)  # outputs "light" or "dark"

# Tmux
tmux source-file ~/.config/tmux/themes/solarized-${THEME}.conf 2>/dev/null

# Neovim — signal via file, nvim watches it
echo "$THEME" > ~/.config/nvim/.current-theme

# VS Code — update settings.json directly (or let extension handle it)
```

### Cron entries (added via chezmoi-managed crontab or launchd plist)

```
0 7  * * *  ~/.local/bin/theme-switch
0 18 * * *  ~/.local/bin/theme-switch
```

## 2. Claude Config in Dotfiles

### What to manage

| File | In dotfiles? | How |
|------|-------------|-----|
| `~/.claude/CLAUDE.md` | Yes | `home/dot_claude/CLAUDE.md` — public, general AI instructions |
| `~/.claude/settings.json` | Yes | `home/dot_claude/settings.json.tmpl` — chezmoi template, machine-role-gated sections |
| `~/.claude/mcp.json` | Probably drop | Settings.json already contains mcpServers; this appears redundant |
| `~/.claude/projects/` | No | Machine-local memory, drifts intentionally |
| `~/.claude/history.jsonl` | No | Private chat history |
| `~/.claude/plugins/`, `skills/`, `cache/` | No | Auto-downloaded |

### Settings.json template strategy

Base settings (public, all machines):
- `model`, `effortLevel`, `enabledPlugins`, `allowedTools`, `env`
- Universal MCP servers: `context7`, `playwright`
- `statusLine` using `{{ .chezmoi.homedir }}` instead of hardcoded path

Work-machine additions (role-gated in template):
```
{{- if eq .role "work" }}
  ... work MCP servers with keys from 1Password ...
{{- end }}
```

1Password template syntax for a secret:
```
{{ onepasswordRead "op://Personal/Claude MCP Keys/some-api-key" }}
```

On macOS: uses desktop app via `op` CLI.
On Linux server: uses `OP_SERVICE_ACCOUNT_TOKEN` env var (set once, stored in server's env or a local `.env` file that chezmoi reads).

### CLAUDE.md content strategy

Keep public content in `home/dot_claude/CLAUDE.md`:
- Coding style preferences
- Workflow instructions (use TDD, prefer small commits, etc.)
- Tool preferences

Keep private/personal context out — or add it to `~/.claude/CLAUDE.local.md` (Claude Code gitignores this automatically) on each machine.

## 3. Work Machine Sync — Capturing Drift

The work machine has its own `~/.claude/settings.json` with extra MCP servers and possibly other Claude config. It also has work-specific shell, git, and SSH config. This is the process to reconcile.

### Step 1: Inventory the work machine

Before applying the dotfiles, capture everything that matters:

```sh
# Claude settings
cat ~/.claude/settings.json

# Shell startup
cat ~/.zshrc ~/.zshenv ~/.zprofile
ls ~/.config/zsh/

# Git
cat ~/.gitconfig ~/.gitconfig.local

# SSH
cat ~/.ssh/config

# PATH and env
echo $PATH
env | grep -i proxy   # corporate proxy?

# Installed tools
which nvim tmux starship atuin zoxide fzf mise direnv
brew list 2>/dev/null | sort
```

### Step 2: Categorize each difference

For every setting that differs from the personal machine, ask:

| Question | If yes → |
|----------|----------|
| Is this useful on personal machine too? | Move into shared dotfiles |
| Is this work-specific but stable (git signing key, work email)? | Add to role-gated template or `~/.gitconfig.local` |
| Is this a secret or credential? | Store in 1Password, inject via chezmoi template |
| Is this company-required PATH or tool? | Put in `~/.config/zsh/local.zsh` (not in repo) |
| Is this a Claude MCP server with an API key? | 1Password entry, injected into `settings.json.tmpl` via role gate |

### Step 3: Apply dotfiles and layer overrides

```sh
# On work machine
chezmoi init --apply https://github.com/adamcooper/dotfiles.git
# Edit role
chezmoi edit-config   # set role = "work"
chezmoi apply

# Work-local shell additions (not committed to repo)
$EDITOR ~/.config/zsh/local.zsh

# Work git identity (already templated, but verify)
cat ~/.gitconfig.local
```

### Step 4: Pull work MCP servers into 1Password

For each work MCP server with a key:

1. Create a 1Password item: `Claude MCP - <service-name>`
2. Add field `api-key` with the value
3. Reference in `settings.json.tmpl`:
   ```json
   "some-service": {
     "command": "npx",
     "args": ["-y", "some-mcp-package"],
     "env": {
       "API_KEY": "{{ onepasswordRead \"op://Work/Claude MCP - some-service/api-key\" }}"
     }
   }
   ```

### Step 5: Consolidation pass

After work machine is running stably:

1. Compare `~/.config/zsh/local.zsh` on both machines — promote anything shared
2. Check tmux and nvim feel consistent
3. Verify theme switching works on both
4. Remove any legacy files from work machine's home dir that the dotfiles now own

## 4. Remote Server as AI Coding Environment

The remote server gets the full treatment — not a stripped-down config. Key differences from Mac:

- **Theme:** time-of-day script still applies; no Ghostty config needed (terminal is the Mac's Ghostty)
- **VS Code:** not on server; nvim is the primary editor
- **Claude:** `settings.json` with role=remote — no work MCPs, but full plugin set and context7/playwright if useful
- **1Password:** install `op` CLI, authenticate with a service account token stored in server's `~/.config/op/service-account-token` (not in repo)

### Claude on remote server

Claude Code runs fine on Linux. The `settings.json` for role=remote should include:
- Same plugins as Mac (superpowers, rails-ai, etc.)
- Universal MCPs only (no work-specific ones)
- No statusLine HUD (unless you set one up server-side)

### Bootstrap sequence for a new remote server

```sh
# 1. Install base packages
curl -fsSL https://raw.githubusercontent.com/adamcooper/dotfiles/master/scripts/bootstrap-linux.sh | bash

# 2. Apply dotfiles
chezmoi init --apply https://github.com/adamcooper/dotfiles.git
chezmoi edit-config   # role = "remote"
chezmoi apply

# 3. 1Password CLI for secret MCPs (optional)
# Install op CLI, then:
export OP_SERVICE_ACCOUNT_TOKEN="..."  # add to ~/.config/zsh/local.zsh
chezmoi apply   # re-apply to render 1Password-sourced values

# 4. Install Claude Code
npm install -g @anthropic-ai/claude-code
```

## Decisions Made

- **Theme switching:** Follow macOS system dark/light preference (not time-of-day)
- **nvim plugin:** `maxmx03/solarized.nvim` — best maintained, proper light+dark, treesitter/LSP support
- **nvim auto-switch:** `auto-dark-mode.nvim` watches system preference, calls `vim.o.background`
- **VS Code:** Native `autoDetectColorScheme` with `preferredLightColorTheme`/`preferredDarkColorTheme`
- **Tmux on macOS:** `dark-mode-notify` (Homebrew) triggers `tmux source-file` on system change
- **Tmux on remote server:** Detect terminal background color via escape sequences (works over SSH from Ghostty)
- **Claude settings.json:** chezmoi template — base public, work MCPs role-gated + 1Password secrets
- **VS Code settings:** managed via chezmoi (not VS Code Settings Sync, to keep it in one place)
